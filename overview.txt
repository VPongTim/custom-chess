1. Core data model

We first create the basic “language” of the game:

Position
Color
Piece
Board
Move

Why: before we can move pieces, draw a board, or add custom rules, the program needs a clean way to represent “where something is” and “what something is.”

2. Board representation

Next we build an 8×8 board that can:

place pieces
remove pieces
get piece at position
print itself in the terminal

Why: this lets us visually check the game state before we have a GUI.

Example:

8 r n b q k b n r
7 p p p p p p p p
6 . . . . . . . .
...
3. Piece hierarchy

Then we create an abstract Piece class and subclasses:

King
Queen
Rook
Bishop
Knight
Pawn

Why: this is where OOP becomes useful. Every piece shares common data, like color and symbol, but each piece has different movement behavior.

4. Basic movement logic

We make pieces generate possible moves.

Example:
Knight on d4 → can move to b3, b5, c2, c6, e2, e6, f3, f5

Why: this teaches inheritance, polymorphism, method overriding, and clean separation of responsibilities.

At first we ignore check/checkmate. Just movement and captures.

5. Game controller

Then we create a Game class that handles:

whose turn it is
whether a move is allowed
moving a piece
capturing
switching turns

Why: the board should not control the whole game. The Game class becomes the “referee.”

6. Terminal playable version

Before graphics, we make a simple terminal version:

Enter move: e2 e4
Enter move: e7 e5

Why: if the terminal version works, the logic is probably solid. The GUI later only has to call existing methods.

7. Special chess rules

Only after basic movement works, we add:

check
checkmate
castling
en passant
pawn promotion
stalemate

Why: these rules are annoying. Adding them too early would slow us down and make the project confusing.

8. Custom pieces

Now we add your fun pieces:

Wizard
Archer
Assassin
Time Knight
piece that changes movement every turn
piece that jumps differently depending on board state

Why: by this point, adding custom pieces should mostly mean creating new Piece subclasses.

9. Simple GUI

Then we build a basic visual board:

8×8 grid
click source square
click target square
show piece symbols
highlight legal moves maybe

Why: the GUI is easier once the rules already work. It becomes a visual layer, not the place where logic is hidden.

10. Polish

Finally:

better visuals
restart button
move history
custom game modes
save/load
GitHub README
maybe screenshots

So the current step, Position, is not random. It is one of the smallest building blocks everything else depends on. After that, Board uses Position; pieces use Position; moves use Position; the GUI eventually translates mouse clicks into Positions.
